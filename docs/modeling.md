# Modeling and Evaluation

PokerLab uses simulation-based supervised learning. The goal is not to solve poker optimally; it is to build a reproducible decision-modeling pipeline under uncertainty.

## Training Data

Training rows come from simulated hands between rule-based agents:

- TAG
- LAG
- Calling Station
- Random

Each decision row contains the 40-feature game-state vector, the action taken, and the final hand result in BB.

## Models

### Hand Strength

Classifier that predicts a bucketed strength label from game-state features.

Current labels are generated from `hand_rank_normalized` as an MVP proxy.

### Opponent Model

Classifier that predicts the action an agent took from a state.

This is useful for behavioral modeling and explaining how features correlate with action selection.

### Bluff Detector

Classifier trained on betting actions. A bet is labeled as a bluff when `hand_rank_normalized < 0.4`.

This is a heuristic label, not ground truth intent.

### EV Model

Regressor that predicts action EV in BB for a `(state, action)` pair. The action is represented with one-hot features appended to the 40 state features.

There are two supported label sources:

- `backend/data/ev_training_data.csv`, generated by counterfactual action rollouts
- fallback `hand_profit_bb`, the final hand result from the action that was actually taken

The counterfactual dataset is preferred. It samples decision states, estimates every legal action with Monte Carlo rollouts, and trains the EV model on those action-specific labels.

This is the main model used for action recommendation.

## Evaluation

Run:

```bash
python3 backend/scripts/generate_ev_data.py
python3 backend/scripts/evaluate_models.py
```

This writes:

```text
backend/artifacts/evaluation_report.json
```

The report contains:

- held-out classification metrics
- EV regression error
- confusion matrices
- feature importances
- seeded agent backtests

## Baselines

The evaluation script includes strategy baselines:

- TAG vs Random
- LAG vs Random
- TAG vs CallingStation
- MLAgent vs TAG, if an EV model is available
- MLAgent vs LAG, if an EV model is available

The main business/product metric is `bb_per_100`, supported by action frequencies, showdown rate, showdown win rate, and bankroll curves.

## Known Limitations

- Final hand profit is an imperfect EV label because it depends on future actions from both agents. Counterfactual rollout labels reduce this problem but are still policy-dependent approximations.
- Rule-based agents create a biased training distribution.
- The engine is heads-up only.
- Bet sizing is bucketed, not continuous.
- Current models are Random Forests for interpretability and implementation speed.

## Strong Next Steps

- Increase counterfactual rollout count and diversify rollout policies for better EV labels.
- Add calibration plots for classifiers.
- Track experiments over multiple random seeds.
- Add confidence intervals for bb/100.
- Add a policy-improvement loop where MLAgent-generated hands become new training data.

## Strategy Optimization

The strategy diagnosis workflow now includes a lightweight policy-search loop:

1. Detect strategy leaks from simulated baseline matchups.
2. Convert leaks into parameter mutations.
3. Backtest candidate strategies across the same baselines.
4. Rank candidates by aggregate score.
5. Return the best policy patch with explicit before/after parameter changes.

This is intentionally pragmatic rather than reinforcement learning. It is a bounded simulation-driven optimizer designed to be understandable, fast enough for product use, and easy to explain in interviews.
